# Skill: skill-reviewer

## 触发条件
用户请求包含以下意图之一：
- “审查这个skill”、“帮我检查skill的质量”
- “评估xxx.md是否合格”、“分析这个skill的问题”
- 用户直接提供一个skill文件内容或链接。

## 审查维度（10项）

| 维度 | 检查点 |
|------|--------|
| 1. 完整性 | 是否包含skill名称、描述、触发条件、工作流程、输出模板？是否缺少必要的错误处理？ |
| 2. 清晰度 | 描述是否无歧义？步骤是否可量化？示例是否易懂？ |
| 3. 自洽性 | 内部逻辑是否矛盾？（例如先说要低成本，后又推荐昂贵方案） |
| 4. 健壮性 | 是否处理用户输入缺失、冲突、超出预期的情况？ |
| 5. 可执行性 | 输出是否可以直接被用户使用（如命令、代码、配置）？还是需要二次加工？ |
| 6. 安全性 | 是否涉及危险操作（如直接修改文件、执行未知脚本）而未加警示？是否包含安全最佳实践？ |
| 7. 可维护性 | 是否模块化？是否方便扩展（如添加新案例）？ |
| 8. 用户体验 | 是否允许用户中途修改？是否有进度提示？是否过于冗长？ |
| 9. 技术时效性 | 推荐的版本、工具是否过时？是否提及LTS或社区活跃度？ |
| 10. 元技能符合度 | 是否符合skill的通用规范（如frontmatter、版本号、作者等）？ |

## 审查流程

1. **接收输入**：获取skill的原始markdown文本。
2. **解析结构**：提取名称、描述、触发条件、步骤、输出等。
3. **逐项检查**：按上述10个维度逐条判断，记录问题和证据。
4. **严重性评级**：高（阻断性）、中（影响体验）、低（优化建议）。
5. **输出审查报告**：包含总体评分（A/B/C/D）、问题列表、修正建议、修正后示例片段。
6. **附加改进建议**：提供可选的增强功能（如增加测试用例、添加版本历史）。
