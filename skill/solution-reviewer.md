# Skill: solution-reviewer

## 触发条件
用户请求包含：
- “帮我审查这个技术方案”、“评估我的架构设计”
- “这个方案有什么问题？”、“我的技术选型合理吗？”
- 用户提供一个方案描述或文档。

## 审查维度

| 维度 | 检查点 |
|------|--------|
| 需求对齐 | 方案是否覆盖了用户明确提出的功能和非功能需求？ |
| 技术选型合理性 | 框架/中间件版本是否过时？是否与团队能力匹配？ |
| 可扩展性 | 未来业务增长时，方案能否低成本扩展？ |
| 安全性 | 是否缺失基础安全措施（认证、授权、加密、防攻击）？ |
| 可维护性 | 代码分层是否清晰？是否有过度设计或欠设计？ |
| 成本效益 | 服务器/人力/运维成本是否与项目规模匹配？ |
| 可操作性 | 是否有具体的实施步骤、命令、配置示例？ |
| 风险与陷阱 | 是否识别了常见技术风险（大事务、缓存穿透、循环依赖等）并提供对策？ |

## 工作流程

1. **解析方案**：提取技术栈、架构模式、模块划分、部署方式等。
2. **逐项检查**：按上述维度评估，记录问题和优势。
3. **严重性评级**：高（可能导致项目失败）、中（影响开发效率或稳定性）、低（优化建议）。
4. **输出审查报告**：包含总体评价、问题清单、改进建议、修正示例。
